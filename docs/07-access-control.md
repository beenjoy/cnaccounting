# 访问控制与权限管理

本文档描述系统的身份认证方式、角色体系、权限划分和成员管理规则，帮助组织管理人员合理配置成员权限。

---

## 一、用户身份认证

### 支持的登录方式

系统支持多种登录方式，用户可以根据组织政策选择适合的方式：

| 登录方式 | 适用场景 |
|---------|---------|
| 邮箱 + 密码 | 标准个人账户登录，适合所有用户 |
| Google 账号 | 使用 Google 账号一键登录 |
| GitHub 账号 | 使用 GitHub 账号一键登录 |
| 企业单点登录（SSO） | 企业统一身份认证，员工使用公司账号登录 |
| 企业 SAML | 使用 SAML 2.0 协议的企业身份系统 |

### 多登录方式绑定

同一用户账号可以同时绑定多种登录方式。例如，用户可以用邮箱密码注册后，再绑定 Google 账号，此后两种方式都可以登录到同一账号。

若用某种新方式登录时，邮箱地址与已有账号相同，系统会自动将两者关联，无需用户手动操作。

### 密码安全要求

- 密码最少 8 个字符
- 密码以加密形式存储，系统人员无法查看明文密码

### 登录会话管理

每次成功登录后，系统建立一个登录会话，有效期因登录方式而异：

| 登录方式 | 会话有效期 |
|---------|----------|
| 邮箱密码 | 7 天 |
| Google / GitHub | 24 小时 |
| 企业 SSO / SAML | 8 小时 |

会话到期后，用户需要重新登录。退出登录时，服务器端的会话记录同步销毁。

## 二、组织角色体系

### 两层角色结构

系统采用**两层角色结构**：每位成员拥有一个**基础角色**（决定管理权限级别），并可额外叠加一个或多个**职能角色**（决定专业操作权限）。

### 基础角色

基础角色决定成员在组织中的管理层级：

| 角色 | 权限级别 | 说明 |
|------|---------|------|
| 所有者（Owner） | 最高 | 完全控制组织的一切内容，不可被其他人移除。通常由创建者或最高负责人担任 |
| 管理员（Admin） | 高 | 拥有绝大多数管理操作权限，不可移除所有者。适合IT管理员或财务团队负责人 |
| 成员（Member） | 中 | 标准业务操作权限，执行日常财务工作 |
| 查看者（Viewer） | 低 | 只读权限，可查看所有数据但不可进行任何修改操作。适合审计员、外部顾问 |

### 职能角色

职能角色叠加在基础角色之上，赋予特定的专业操作权限。同一成员可同时拥有多个职能角色：

| 职能角色 | 对应职位 | 核心权限 |
|---------|---------|---------|
| 会计总监（Controller） | 财务总监、会计主管 | 审批和过账凭证；关闭和重开会计期间；执行年末结账；查看审计日志 |
| 财务经理（Finance Manager） | 财务经理、主计长 | 管理科目表（创建、修改、停用科目）；管理汇率；审批和过账凭证 |
| 会计（Accountant） | 会计、出纳 | 创建凭证；提交凭证审批 |
| 期间管理员（Period Admin） | 财务主管、期间负责人 | 开放和关闭会计期间 |
| 合并经理（Consolidation Manager） | 合并专员、集团财务 | 管理合并组配置；执行合并报表编制 |

### 典型岗位权限配置

| 实际岗位 | 建议基础角色 | 建议职能角色 |
|---------|------------|-----------|
| 集团 CFO | 所有者 或 管理员 | 会计总监 |
| 财务经理 | 成员 | 财务经理 + 会计总监 |
| 会计人员 | 成员 | 会计 |
| 期间控制专员 | 成员 | 期间管理员 |
| 合并报表专员 | 成员 | 合并经理 |
| 外部审计员 | 查看者 | （无需职能角色） |
| 内部审计 | 成员 | 会计总监（用于查看审计日志） |

## 四、自定义权限策略

### 策略的作用

除了内置的角色权限之外，组织管理员可以创建自定义策略，对特定场景进行精细化权限控制：
- **允许策略（Allow）**：允许某角色或某用户在特定条件下执行特定操作
- **拒绝策略（Deny）**：禁止某角色或某用户执行特定操作，即使其基础角色本应有此权限

**示例场景：**
- "只允许会计人员在工作日工作时间内创建凭证"
- "禁止某特定用户过账超过一定金额的凭证"
- "将某位成员的凭证创建权限限制在 A 公司，不允许操作 B 公司"

### 策略配置的四个要素

创建一条策略时，需要配置：

**1. 谁（主体）**
- 适用的基础角色（所有者/管理员/成员/查看者）
- 适用的职能角色（会计总监/财务经理/会计/期间管理员/合并经理）
- 或指定特定用户

**2. 做什么（操作）**
- 选择一种或多种操作（如创建凭证、过账、关闭期间等）

**3. 对什么（资源范围）**
- 选择适用的资源类型（如凭证、科目、公司等）
- 可进一步限定范围：如仅对特定公司生效

**4. 效果**
- 允许：策略匹配时授予权限
- 拒绝：策略匹配时撤销权限（拒绝优先于允许）

**可选高级条件：**
- 时间限制：如仅工作日（周一至周五）的工作时间（9:00–18:00）生效

### 策略优先级规则

当多条策略适用于同一操作时：
1. 拒绝策略优先于允许策略
2. 系统内置的保护策略拥有最高优先级，不可被自定义策略覆盖
3. 同等优先级下，具体策略（特定用户）优先于通用策略（角色范围）

### 策略配置入口

策略管理有两个入口：

| 入口 | 位置 | 适用场景 |
|------|------|---------|
| 策略管理页 | 组织设置 → 权限策略 | 管理适用于特定角色的组织级通用规则 |
| 成员详情页 | 组织设置 → 成员 → 某成员 | 管理针对特定用户的个性化权限例外 |

## 五、系统内置保护策略

以下四条保护策略由系统内置，永远不可删除或被自定义策略覆盖：

| 策略名称 | 效果 | 保护内容 |
|---------|------|---------|
| 已关闭期间保护 | 拒绝 | 已关闭的会计期间不允许任何人过账凭证，包括管理员 |
| 有效科目要求 | 拒绝 | 已停用的科目不允许在凭证中使用 |
| 借贷平衡验证 | 拒绝 | 借贷不平衡的凭证不允许过账 |
| 所有者不可移除 | 拒绝 | 拥有者角色的成员不允许被非拥有者移除 |

## 六、成员管理

### 成员状态

组织成员有三种状态：

| 状态 | 说明 |
|------|------|
| 活跃 | 正常状态，可以登录和操作 |
| 已停用 | 账号被暂时停用，无法登录，但历史记录保留。可恢复为活跃状态 |
| 已移除 | 已从组织中移除，无法再访问组织数据。历史记录保留用于审计追溯 |

### 成员邀请

- 管理员向指定邮箱地址发送邀请，同时设定受邀人的角色
- 邀请链接有效期通常为 7 天
- 受邀人接受邀请后自动成为组织成员
- 若受邀邮箱无系统账号，引导完成注册后自动加入
- 同一邮箱对同一组织同时只能有一条待接受的邀请

### 成员停用与移除

- **停用**：临时取消成员访问权限，成员无法登录，但数据保留，可日后恢复
- **移除**：永久解除与组织的关联，移除后不可恢复为正常成员
- **特别保护**：所有者角色的成员不可被非所有者移除

### 角色变更

管理员可随时调整活跃成员的基础角色和职能角色，变更立即生效。所有角色变更操作均记录在审计日志中。

## 七、数据隔离与多组织

### 组织间完全隔离

- 每个组织的所有数据完全隔离，不同组织之间无法互相访问
- 用户只能看到自己所属组织的数据
- 即使同一用户属于多个组织，各组织的数据也独立存储，互不干扰

### 多组织切换

一个用户可以属于多个组织，在每个组织中可以拥有不同的角色。用户可以在各组织之间切换，但同一时刻只能操作一个组织的数据。

### 跨组织访问限制

任何跨组织的数据访问均被系统严格禁止。尝试访问所属组织以外的数据，系统将拒绝请求并返回权限错误。

---

*本文档描述了系统完整的访问控制体系。建议组织管理员在分配权限时遵循"最小权限原则"——只授予成员完成工作所必需的最低权限。*
